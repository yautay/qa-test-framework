## Ticket Synopsis

FEATURE-001 adds authentication support (Basic Auth and Bearer Token) for connections to WS Grid (Docker Playwright and Selenium Grid). Both `grid_ws_endpoint` and `grid_cdp_endpoint` need independent auth modes (`none` | `basic` | `token`). Credentials are stored in `settings.py`, overridable by env vars (env takes priority), and must be masked in all outputs. Auth headers are injected at connect time only (no pre-validation). 401 errors must surface as a clear per-test `GridAuthError`.

## Summary

All research questions from the ticket are answered:

1. **Playwright `connect()` and `connect_over_cdp()` both accept a `headers` kwarg** (confirmed in installed Playwright 1.51.0). No workaround needed.
2. **`RuntimeEnv` is a `@dataclass(frozen=True)`** with no `__repr__`/`__str__` override — the default dataclass repr will expose all field values. Masking must be added explicitly.
3. **`env_str` pattern is fully established** — 8 new fields follow the exact same 2-line pattern used by all existing fields.
4. **Jira `client.py` has a complete Basic Auth via `base64` reference implementation** — `_auth_header()` covers both `basic` and `token` modes using stdlib `base64.b64encode`.
5. **`http_sender.py` has a complete Bearer token reference** — `_build_headers()` sets `Authorization: Bearer {token}`.
6. **HTTP probe requests in `session.py`** (`requests.post` at line 235, `requests.get` at line 289, `requests.delete` at line 311) currently only use `Content-Type` — auth headers must be added there too.
7. **No masking mechanism currently exists** in `RuntimeEnv` or elsewhere for sensitive values.
8. **`settings_cli.py` has no GRID fields at all** — confirmed no changes needed there.

## Detailed Findings

### 1. RuntimeEnv — structure and masking gap

`framework/env.py:60-153`

- `RuntimeEnv` is `@dataclass(frozen=True)`.
- No `__repr__`, `__str__`, or `__post_init__` defined.
- Python's default `dataclass.__repr__` includes **all fields**, meaning `grid_ws_password`, `grid_ws_token` etc. would be fully visible in any log that prints the env object.
- **Existing sensitive fields already unmasked**: `jira_password` (line 145), `jira_api_token` (line 148), `visual_minio_secret_key` (line 122), `reporting_api_token` (line 86) — none are masked today.
- **Masking approach**: Add `__repr__` (or `__str__`) to `RuntimeEnv` that redacts known secret fields. Alternatively use a `field(repr=False)` annotation (available on dataclass fields in Python 3.10+).
- **Recommended**: use `field(repr=False)` for each secret field — simpler, no custom repr needed, and auto-excluded from `repr()` output.

```python
# Example for new fields:
from dataclasses import dataclass, field

grid_ws_password: str = field(default="", repr=False)
grid_ws_token: str    = field(default="", repr=False)
```

However, `RuntimeEnv` is frozen and uses positional field declaration without defaults — all fields are currently required positional args in the constructor. The `field(repr=False)` approach works fine alongside positional fields (no default needed), e.g.:

```python
grid_ws_password: str = field(repr=False)
```

This masks the value in `repr()` but still requires it to be passed at construction time.

### 4. Auth header builder — helper function

Based on Jira `_auth_header()` pattern (`framework/integrations/jira/client.py:50-64`):

```python
import base64

def _build_grid_auth_headers(auth_mode: str, username: str, password: str, token: str) -> dict[str, str]:
    mode = (auth_mode or "none").strip().lower()
    if mode == "basic":
        user = (username or "").strip()
        secret = (password or "").strip()
        if user and secret:
            raw = f"{user}:{secret}"
            encoded = base64.b64encode(raw.encode("utf-8")).decode("ascii")
            return {"Authorization": f"Basic {encoded}"}
    elif mode == "token":
        tok = (token or "").strip()
        if tok:
            return {"Authorization": f"Bearer {tok}"}
    return {}
```

Jira uses **identical** `base64.b64encode` logic (line 63). The `http_sender.py` uses `f"Bearer {self.token}"` (line 62). Both patterns are proven in the codebase.

### 8. Masking scope — all serialization paths

The ticket requires masking in:
- Tool logs (loguru `logger.*` calls)
- Allure, pytest-html
- Jira attachments

**Root cause**: If any code does `str(runtime_env)` or `repr(runtime_env)` or passes env fields directly to loguru structured args, secrets leak.

**Recommended fix**: Use `dataclasses.field(repr=False)` on the 4 sensitive fields (`grid_ws_password`, `grid_ws_token`, `grid_cdp_password`, `grid_cdp_token`). This prevents them from appearing in `repr()` output (which is what loguru and most serializers call).

Note: `jira_password` and `jira_api_token` and `visual_minio_secret_key` and `reporting_api_token` are **also currently unmasked** in `RuntimeEnv`. The ticket scope is limited to grid auth fields, but the same `field(repr=False)` fix could be applied to those too as a bonus.

## Architecture Insights

1. **`env_value` priority chain** (env.py:161-166): `os.environ` → `.env` dotenv → `None`. All `env_str` calls inherit this. New auth fields follow the same chain automatically.

2. **`RuntimeEnv` is a frozen dataclass** — immutable once constructed by `load_env()`. All new fields must be declared in the class body AND passed in the `return RuntimeEnv(...)` call.

3. **`getattr(settings, "field_name", default)` guard** (env.py:472+) — used for newer fields added to `settings.py` after initial release. The same pattern should be used for grid auth fields to avoid `AttributeError` if someone runs without the new `settings.py`.

4. **Playwright `headers` kwarg** applies to the HTTP/WebSocket upgrade request — exactly the right layer for `Authorization: Basic` or `Authorization: Bearer`. The header is sent once at connection establishment, not on subsequent WebSocket frames.

5. **`_connect_auto_grid`** (session.py:84-110) calls both `_connect_playwright_grid` and `_connect_selenium_cdp_grid` internally — auth injection in those functions is sufficient, no changes needed in `_connect_auto_grid` itself.

6. **HTTP probes (`requests.post/get/delete`) in session.py** target the same Grid host as `grid_ws_endpoint`. Therefore they should use the **WS auth credentials**, not CDP auth credentials (CDP auth is for the final WebSocket CDP connection, not the Selenium REST API).

7. **Error wrapping**: For the Playwright WS connect, auth errors manifest as connection exceptions (not HTTP 401 directly). A safe detection strategy: catch `Exception` from `browser_type.connect(...)`, check if the message contains "401" or "Unauthorized", then raise `GridAuthError`. For HTTP probes, check `response.status_code == 401` explicitly.

## Open Questions

1. **Masking of pre-existing sensitive fields** (`jira_password`, `jira_api_token`, `reporting_api_token`, `visual_minio_secret_key`) — **DECISION: yes, mask these in the same PR** using `field(repr=False)`. Trivial change, same mechanism.

2. **`_delete_selenium_session` auth** — does DELETE to Selenium Grid also require auth? Likely yes (same Grid host), but if the session already has an auth cookie from the POST, it may not. Safe assumption: add headers to DELETE too.

3. **CDP endpoint auth vs WS auth** — when `grid_cdp_endpoint` is set explicitly (not discovered from Selenium Grid), the `connect_over_cdp()` call uses `grid_cdp_auth_mode`. But the HTTP probe requests (`_create_selenium_session_for_cdp`, `_try_get_grid_cdp_endpoint`) always go to `grid_url` (derived from `grid_ws_endpoint`). So WS auth creds should be used for HTTP probes regardless of CDP auth config.

4. **Token auth for Jira uses `username:api_token` as Basic Auth** (not Bearer) — see `jira/client.py:56-64`: `token` mode still uses `base64(username:api_token)` format. For Grid, `token` mode should use `Bearer <token>` (without username), which is different. This is correct per ticket spec but worth noting the difference from Jira's approach.
