
=== API i mocki MUST HAVE ===
Bruno - Kontekst: utrzymanie kolekcji testów API w repozytorium oraz ich przegląd w procesie code review. Uzasadnienie: wersjonowanie przypadków testowych razem z kodem zapewnia pełną śladowość zmian, spójność procesu testowego i możliwość audytu. Ryzyko braku: rozproszenie testów API, brak historii zmian oraz trudność w odtworzeniu zakresu weryfikacji wykonanej przed wdrożeniem.
Mockoon - Kontekst: testy integracyjne w sytuacji niedostępności lub ograniczonego dostępu do usług partnerów i systemów zewnętrznych. Uzasadnienie: umożliwia tworzenie kontrolowanych mocków usług bez przetwarzania realnych danych zewnętrznych i bez ryzyka naruszenia ustaleń integracyjnych. Ryzyko braku: blokada części testów, opóźnienia wdrożeń oraz wzrost ryzyka prowadzenia testów na środowiskach partnerów w sposób niekontrolowany.
Postman - Kontekst: regresja endpointów, walidacja autoryzacji, testy kontraktów oraz ręczne i półautomatyczne scenariusze API. Uzasadnienie: standaryzuje sposób weryfikacji interfejsów i ogranicza ryzyko publikacji usług niezgodnych z umową integracyjną. Ryzyko braku: wzrost liczby defektów integracyjnych po wdrożeniu oraz większa liczba incydentów po stronie klienta lub systemów zależnych.
SoapUI - Kontekst: testy API SOAP/REST, walidacja WSDL/XSD oraz scenariusze integracji między systemami, w szczególności testy integracyjne SKLEP <-> SEZAM. Uzasadnienie: narzędzie jest niezbędne w testach integracyjnych SKLEP <-> SEZAM, ponieważ umożliwia formalną weryfikację kontraktów usług i testy regresji na poziomie komunikatów, co jest kluczowe dla stabilnej wymiany danych między systemami. Ryzyko braku: wyższe ryzyko błędów interoperacyjności, niezgodności kontraktów oraz opóźnień we wdrożeniach integracyjnych.

=== Bazy danych MUST HAVE ===
DBeaver - Kontekst: codzienna diagnostyka danych w wielu silnikach bazodanowych podczas testów, analiz powdrożeniowych oraz obsługi incydentów. Uzasadnienie: jedno narzędzie wielosilnikowe ogranicza ryzyko błędów wynikających z różnic między klientami baz danych i skraca czas reakcji operacyjnej. Ryzyko braku: wydłużona analiza danych, większa liczba pomyłek przy ręcznej walidacji oraz opóźnienia w obsłudze zgłoszeń.
DataGrip - Kontekst: praca na złożonych zapytaniach, analiza schematów i refaktoryzacja SQL. Uzasadnienie: zaawansowane podpowiedzi i inspekcje statyczne ograniczają ryzyko wykonania wadliwych zapytań na danych roboczych i testowych. Ryzyko braku: wzrost ryzyka naruszenia integralności danych oraz wyższy koszt przygotowania i przeglądu zmian SQL.
HeidiSQL - Kontekst: operacyjne testy panelu sprzedawcy oraz szybkie potwierdzanie stanu rekordów i skutków wykonanych operacji. Uzasadnienie: lekkie narzędzie do szybkiej weryfikacji i porównywania danych umożliwia sprawną obsługę zgłoszeń funkcjonalnych i incydentów. Ryzyko braku: dłuższy czas diagnozy błędów oraz niższa skuteczność wsparcia operacyjnego.
MySQL Workbench (SQL Workbench) - Kontekst: administracja i audyt konfiguracji MySQL oraz testy edycji danych tabelarycznych. Uzasadnienie: wspiera kontrolowaną walidację struktur, uprawnień i procesu zapisu zmian, co jest istotne dla rozliczalności działań administracyjnych i testowych. Ryzyko braku: utrudniony audyt zmian konfiguracyjnych oraz wyższe ryzyko nieautoryzowanych lub błędnych modyfikacji.
pgAdmin - Kontekst: diagnostyka, administracja i testy danych w środowiskach PostgreSQL. Uzasadnienie: zapewnia bezpieczną i audytowalną obsługę baz PostgreSQL oraz szybką weryfikację struktur, indeksów i danych wykorzystywanych przez aplikacje i testy integracyjne. Ryzyko braku: utrudniona obsługa środowisk opartych o PostgreSQL, wolniejsza diagnostyka oraz większe ryzyko pomyłek przy ręcznej analizie danych.

PyCharm - Kontekst: utrzymanie narzędzi testowych i automatyzacji w Pythonie. Uzasadnienie: integracja test runnerów i analizy statycznej poprawia jakość zmian oraz ich powtarzalność. Ryzyko braku: trudniejsze utrzymanie testów, niższa jakość kodu i wydłużony czas debugowania.
WebStorm - Kontekst: rozwój i utrzymanie aplikacji frontendowych JavaScript/TypeScript. Uzasadnienie: narzędzia analizy kodu i integracja z testami ograniczają ryzyko regresji w warstwie interfejsu użytkownika. Ryzyko braku: późniejsze wykrywanie błędów i większa liczba poprawek po wdrożeniu.
Zed - Kontekst: awaryjna i szybka edycja kodu przy ograniczonych zasobach lub czasowej niedostępności głównych IDE. Uzasadnienie: zapewnia ciągłość pracy i możliwość natychmiastowej interwencji technicznej. Ryzyko braku: przestoje zespołu i wydłużony czas obsługi incydentów krytycznych.

=== Narzędzia developerskie MUST HAVE ===
Docker Desktop - Kontekst: uruchamianie lokalnych kontenerów aplikacji, baz danych i usług pomocniczych w ramach codziennego toolingu deweloperskiego i testowego. Uzasadnienie: absolutnie niezbędny element toolingu, ponieważ zapewnia powtarzalne, izolowane i zgodne ze środowiskiem CI/CD uruchamianie komponentów, co bezpośrednio redukuje różnice między dewelopmentem, testem i produkcją. Ryzyko braku: bez Docker Desktop zespół nie może efektywnie używać kluczowych narzędzi deweloperskich i testowych opartych o kontenery, co blokuje realizację testów, utrzymanie środowisk oraz bieżącą pracę projektową.
Git - Kontekst: wersjonowanie kodu, historii zmian i zarządzanie wydaniami. Uzasadnienie: zapewnia rozliczalność autorstwa, możliwość audytu oraz bezpieczny rollback zmian. Ryzyko braku: bez Git zespół nie może używać podstawowych narzędzi deweloperskich i testowych opartych o repozytoria, branche oraz pipeline CI/CD, co blokuje codzienną pracę projektową.
Git Bash - Kontekst: lokalna praca z repozytoriami, skryptami shellowymi i narzędziami CLI w środowisku Windows. Uzasadnienie: zapewnia zgodny z praktykami developerskimi interfejs bash do obsługi Git, automatyzacji i skryptów wykorzystywanych w procesie dev/test. Ryzyko braku: bez Git Bash zespół ma ograniczoną możliwość uruchamiania standardowych skryptów developerskich i testowych, co spowalnia pracę i utrudnia utrzymanie zgodności procesowej.
GitKraken - Kontekst: wizualne zarządzanie branchami, konfliktami i procesem review. Uzasadnienie: standaryzuje pracę zespołu o zróżnicowanych kompetencjach i redukuje błędy przy integracji kodu. Ryzyko braku: większa liczba konfliktów merge, wyższy koszt onboardingu i wolniejsza integracja zmian.
JetBrains Toolbox - Kontekst: centralne zarządzanie instalacją i aktualizacjami IDE. Uzasadnienie: utrzymuje zgodność wersji narzędzi w zespole i ogranicza ryzyko wynikające z nieaktualnego oprogramowania. Ryzyko braku: rozjazd wersji, problemy kompatybilności oraz luki bezpieczeństwa.
Node.js - Kontekst: uruchamianie narzędzi build, test i automatyzacji w projektach frontend oraz full-stack. Uzasadnienie: krytyczny runtime dla CI/CD i lokalnej walidacji artefaktów przed publikacją. Ryzyko braku: bez Node.js zespół nie może używać kluczowych narzędzi dev i test, takich jak build, lint, testy i skrypty automatyzacji, co uniemożliwia przygotowanie i weryfikację wydania.
Windows Terminal - Kontekst: codzienna praca z CLI, skryptami i sesjami wielośrodowiskowymi. Uzasadnienie: standaryzuje procesy administracyjne i zmniejsza ryzyko błędów manualnych przy operacjach technicznych. Ryzyko braku: bez terminala zespół nie może efektywnie używać narzędzi dev i test opartych o CLI, co utrudnia uruchamianie skryptów, testów i zadań operacyjnych.
WSL2 + Ubuntu 22 LTS - Kontekst: lokalne środowisko zbliżone do serwerów Linux. Uzasadnienie: ogranicza różnice między dewelopmentem a produkcją i poprawia wiarygodność wyników testów. Ryzyko braku: bez WSL2 + Ubuntu zespół nie może używać części kluczowych narzędzi dev i test wymagających środowiska Linux, co blokuje pełną walidację i zwiększa ryzyko błędów po wdrożeniu.
XAMPP (PHP, MySQL) - Kontekst: lokalne uruchamianie aplikacji PHP i baz MySQL w scenariuszach developerskich, serwisowych i testowych. Uzasadnienie: pozwala szybko odtworzyć kompletne środowisko aplikacyjne na stacji roboczej bez oczekiwania na wspólne zasoby, co przyspiesza debugowanie i weryfikację zmian. Ryzyko braku: utrudnione lokalne odtwarzanie błędów, wydłużony czas diagnozy oraz blokady w testach wymagających pełnego stosu PHP/MySQL.
